# v0.9 Multi-Organism Resolver — Design Spec

> Date: 2026-05-23
> Status: approved-for-plan
> Author: brainstorming session, plant-genomics-mcp

## Goal

Broaden `plant-genomics-mcp` beyond Arabidopsis (and the handful of partially-supported species) to handle a curated set of ~12 plant organisms across all backends, with a single user-facing `organism=` parameter that accepts slug, scientific name, common name, or NCBI taxid.

## Motivation

As of v0.8.1, every backend tool takes its own organism-ID format:

- `ensembl_plants`, `gramene`, `atted_ii`, `europe_pmc` slug-strip — Ensembl species slug (e.g. `arabidopsis_thaliana`)
- `uniprot`, `string`, `blast` — NCBI taxid int (e.g. `3702`)
- `phytozome` — Phytozome integer organism ID (e.g. `167`)

The model has to know the format per backend, and `resources.py:71` flags that "only arabidopsis_thaliana (167) is controller-verified" — meaning the other 9 entries in `phytozome.KNOWN_ORGANISMS` were never cross-verified against the live backend. v0.9 unifies the input surface and ships a coverage matrix that's verified live on release.

## Non-goals

- Universal plant coverage. The curated table is ~12 plants; adding more is an additive release task, not a v0.9 deliverable.
- Live discovery of organisms (no API call on each `resolve()`). The mapping is hardcoded; drift is caught by `scripts/verify_organisms.py` re-running pre-release.
- Backwards-compatible `species=` aliasing. Clean break per v0.x convention.

## Design

### Data model

New module `src/plant_genomics_mcp/organisms.py`:

```python
@dataclass(frozen=True)
class OrganismRecord:
    canonical: str              # snake_case slug, e.g. "arabidopsis_thaliana"
    scientific: str             # "Arabidopsis thaliana"
    common: tuple[str, ...]     # ("thale cress", "mouse-ear cress")
    ncbi_taxid: int             # 3702
    ensembl_slug: str | None    # "arabidopsis_thaliana" — None if not in Ensembl Plants
    phytozome_int: int | None   # 167 — None if not in Phytozome
    string_taxid: int | None    # 3702 — usually mirrors ncbi_taxid; None if STRING lacks coverage
    europe_pmc_slug: str | None # used for AT-style ID stripping; None means "use ID as-is"
    aliases: tuple[str, ...]    # extra normalized forms accepted as input
```

`ORGANISMS: dict[str, OrganismRecord]` keyed by `canonical`.
`_ALIAS_INDEX: dict[str, str]` maps every normalized input form to `canonical`. Built once at module import.

Backend ID fields are `| None` so coverage gaps are explicit at the data layer.

### Resolver surface

```python
ORGANISMS: dict[str, OrganismRecord]

def resolve(query: str | int) -> OrganismRecord: ...
def ensembl_slug_for(query: str | int) -> str: ...
def phytozome_int_for(query: str | int) -> int: ...
def ncbi_taxid_for(query: str | int) -> int: ...
def string_taxid_for(query: str | int) -> int: ...
def europe_pmc_slug_for(query: str | int) -> str | None: ...

DEFAULT_ORGANISM = "arabidopsis_thaliana"  # canonical key
```

`resolve()` normalization:

- `str.lower().strip()`
- spaces ↔ underscores ↔ hyphens
- scientific abbrev expansion (`A. thaliana` → `arabidopsis thaliana`)
- `int` input treated as `ncbi_taxid`, looked up via reverse index

Per-backend helper behavior:

- ID present → return it
- ID is `None` → raise `OrganismNotSupported(backend, organism, supported_list)`
- `europe_pmc_slug_for` returns `None` directly (matches existing `europe_pmc.py:130` contract for AT-style IDs)

### Exception hierarchy

Two new classes in `errors.py`, both extending the existing `PlantGenomicsError` base (sibling of `RateLimitError`, `NotFoundError`, `UpstreamUnavailableError`):

```python
class OrganismNotFound(PlantGenomicsError):
    """Input didn't match any record in ORGANISMS."""

class OrganismNotSupported(PlantGenomicsError):
    """Input resolved, but the requested backend has no ID for it."""
```

`PlantGenomicsError.__str__` already prepends `[<ClassName>] ` for non-base subclasses, so both classes stringify into the envelope's existing `[ExceptionClass] message` wire format with no changes to `StepRow` or `SynthesisEnvelope`.

### Tool integration

Every backend tool takes a single `organism: str | int = DEFAULT_ORGANISM` param. The old `species=` and `organism_id=` are dropped.

```python
# Ensembl-flavor:
slug = organisms.ensembl_slug_for(organism)
# NCBI-flavor:
taxid = organisms.ncbi_taxid_for(organism)
# Phytozome-flavor:
phyto_id = organisms.phytozome_int_for(organism)
```

server.py schemas all change to:

```python
"organism": {
    "type": ["string", "integer"],
    "default": "arabidopsis_thaliana",
    "description": "Plant organism — accepts slug, scientific/common name, or NCBI taxid",
}
```

`DEFAULT_SPECIES` is currently defined twice (`synthesis.py:43` and `prompts.py:40`); both are consolidated into a single `DEFAULT_ORGANISM` exported from `organisms.py`, then imported at the 5 use-sites (synthesis.py:152, 447, 698; prompts.py:63, 212). Net change: 2 defs deleted, 1 def added, 5 imports added.

### Synthesis tool behavior

Phase-1 resolution happens once at the orchestrator entry; `OrganismNotFound` triggers existing root-failure semantics (envelope ships with `steps[0].status="error"`, all phase-2 steps `status="skipped"` with `error="phase 1 failed"`, `result=None`).

Phase-2 per-backend calls are wrapped: `OrganismNotSupported` → `StepRow(status="skipped", error="[OrganismNotSupported] <msg>")`. Other exceptions still flow through the existing envelope error path. Synthesis composes a result from the backends that succeeded.

### Implementation passes

- **Pass 1** (early task): scaffold `OrganismRecord` + `organisms.py` with Arabidopsis only. All existing default behavior preserved. Green-baseline before any other changes.
- **Pass 2** (later task): run `verify_organisms.py`, populate other 11 records, commit verified matrix.

### Backward compatibility

- v0.x convention: clean breaks at minor-version bumps. v0.7 added tools, v0.8 changed return shape, v0.9 renames params.
- No `species=` / `organism_id=` aliasing.
- Default value (`arabidopsis_thaliana`) unchanged, so callers that never set the param keep working.
- CHANGELOG `## [0.9.0]` with a `### Breaking` block listing every renamed param.
- README "Migration" subsection: `s/species=/organism=/g; s/organism_id=/organism=/g`.
- prompts.py docstrings updated.
- `resources.py:71` rewritten: "Curated coverage matrix: 12 plants, verified live on each release. See `organisms.py:ORGANISMS` for the full list." Static MCP resource exposes the matrix as a table.

## Testing

Three layers, mirroring project convention.

**Unit tests** — `tests/test_organisms.py` (new):

- `resolve()` happy path per record (slug / scientific / common / abbrev / taxid int)
- Case + whitespace + underscore + hyphen normalization
- `OrganismNotFound` raised with non-empty `supported` list
- Each per-backend helper: returns ID when present, raises `OrganismNotSupported` when `None`
- Exception messages include the supported list

**Integration tests** — modify existing `tests/test_*.py`:

- One test per backend tool confirming `organism="oryza_sativa"` resolves and reaches the backend with the right ID format (mock httpx)
- One test per synthesis tool confirming phase-2 `OrganismNotSupported` surfaces as `status="skipped"` in the envelope
- Phase-1 `OrganismNotFound` produces existing root-failure envelope

**Live tests** — extend existing `PLANT_GENOMICS_MCP_LIVE=1` gating:

- Per backend, one live call against a non-Arabidopsis organism confirming the resolved ID is accepted upstream (real-execution check per project doctrine)
- `scripts/verify_organisms.py` doubles as the pre-release verification harness
